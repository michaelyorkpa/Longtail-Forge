# 0.32 Module and File Framework Closeout

Version 0.32.13 closes the 0.32 framework work before the Notes module begins. The review below records the current module boundary and file readiness checks so later 0.33.x work can consume the existing contracts instead of adding new framework primitives by accident.

## Module Isolation Review

- First-party workflow records remain module-owned. Clients/projects, tasks, time entries, tags, and users keep their own manifests, services, routes, and storage responsibilities.
- Framework-owned services remain generic for permissions, tags, search, notifications, audit logging, Help, files, events, app shell navigation, settings discovery, and protected static page routing.
- Timers are not owned by Tasks. Task timers use the task source contract and finalize through Time Tracking into normal time entries.
- Module manifests remain the primary integration point for navigation, settings, permissions, API scopes, audit record types, event types, event summaries, notification declarations, taggable types, searchable types, attachable file targets, Help contributions, Workbench cards, timer sources, and work-item sources.
- Disabled-module behavior is filtered through framework contribution discovery and route guards. Modules should not add browser-only visibility checks as substitutes for service/API permission checks.

## File Framework Readiness

- Supported modules can attach files by declaring `attachableTypes` and using the framework file service and shared browser helper.
- Future Notes, Support Tickets, Knowledge Base, Creator Studio, Lists, Projects, Clients, and Calendar surfaces should be able to consume the current file contract with module-owned placement and wording.
- Public/client-visible file behavior must use explicit file visibility values and permission checks. Tags remain classification metadata and must not become the source of truth for public/private access.
- File lifecycle events are safe integration points for notifications, audit logging, search refreshes, activity summaries, and local module UI refreshes. Payloads must not include raw file contents, unsafe storage paths, secrets, or scanner internals.
- The no-op scanner remains appropriate only for the current trusted/admin-oriented upload surface. Broad public or user-generated uploads still require a real safety/scanning policy before expansion.

## 0.33.5.4 File QoL Closeout

- Files deletion is staged. Normal delete marks a file deleted, shows an in-app warning first, preserves safe attachment/history rows, and leaves protected bytes in place for the retention window. Restore reopens the soft-deleted file; hard purge remains a separate admin-managed lifecycle.
- Uploads use Files-owned server-side type policy checks. Multi-file uploads return per-file success/failure results so accepted files stay attached when another selected file is rejected.
- Files workspace settings own file type policy, reserved storage limits, and aggregate storage accounting. General workspace settings remain responsible for workspace identity and unrelated defaults.
- Storage accounting is aggregate-only and permission-gated. Internal protected bytes and external provider-reported bytes are tracked separately, and accounting payloads must not expose file labels, attachment target labels, storage keys, protected paths, signed URLs, scanner internals, or per-file metadata.

## 0.33.5.18.11 Browse/Edit/Preview Closeout

- The Files page is a compact workspace file recovery surface, not document management. The filter sidebar owns browse filtering, and the main panel owns the listing only.
- The 0.33.5.18.11.4 inline detail/summary/preview anatomy was intentionally replaced. Normal browse must not show inline Browse Summary, selected-file headers, inline Metadata panels, inline Preview panels, selected-row detail state, or Inspector integration.
- Row click opens the File Context edit modal. File metadata is read-only in that modal, and context edits are attachment-scoped and route-backed through the Files service.
- Preview opens the Files Preview modal. Preview is attachment-scoped and route-backed through the preview descriptor/content routes rather than a persistent browse pane.
- Download, Delete, Restore, Report, and future Quarantine actions remain distinct route-backed Files actions. The next upload/action slices may standardize their controls and placement, but should not reimplement File Context or Preview routes/modals.
- Rename, file replacement, storage moves, hard purge, permanent delete, raw storage controls, and unsafe direct metadata editing remain out of scope unless the Files service ships explicit routes, permissions, audit behavior, and regressions for them first.

## Help and Documentation Review

- Help Center is the in-app user guidance surface for current behavior. It now covers Tasks, Time Tracking, Clients and Projects, Notifications, Tags, Search, Files and Attachments, settings, workspaces, users, and modules.
- Developer documentation records attachable manifest declarations, file lifecycle event payload rules, storage/scanner adapter boundaries, the shared browser attachment helper contract, and module consumption rules.
- Roadmap-level future promises stay in `ROADMAP.md`; Help articles should remain task-oriented and current-state focused.

## 0.33 Readiness Notes

- Notes should be built as a first-party module with its own records, permissions, routes, search declarations, tag declarations, file attachable target, Help contributions, and module lifecycle behavior.
- Notes should use the existing framework services for tags, search, notifications, audit logging, files, events, permissions, and Help.
- Notes should not require new file primitives unless implementation reveals a concrete gap in the existing attachable target, browser helper, lifecycle event, or permission contract.
